# AI 语音助手 - 项目介绍

> **Sound AI Assistant** — 一个基于浏览器原生 API 的智能语音对话助手，支持语音输入、AI 流式对话、边生成边朗读、知识自动提取，数据完全本地存储。

---

## 一、项目概述

### 1.1 这是什么项目？

这是一个**纯前端**的 AI 语音助手应用，核心能力是让用户通过**语音**与 AI 进行自然对话：

- **语音输入**：对着麦克风说话，自动转成文字发送给 AI
- **流式对话**：AI 回复边生成边显示（打字机效果）
- **边生成边朗读**：AI 回复生成过程中就按句子实时朗读，不用等全部生成完
- **知识自动提取**：从对话中自动识别日程、待办、笔记、联系人等信息
- **本地持久化**：所有对话和知识存储在浏览器 IndexedDB 中，无需后端

### 1.2 技术栈

| 分类 | 技术 | 说明 |
|------|------|------|
| 框架 | React 18 + TypeScript | 严格类型安全 |
| 构建 | Vite 5 | 极速 HMR，ESM 原生支持 |
| 样式 | Tailwind CSS 3 | 原子化 CSS，自定义主题色 |
| 状态管理 | Zustand 4 | 轻量、灵活、无 Provider |
| 本地数据库 | Dexie 4 (IndexedDB) | 结构化存储，支持索引查询 |
| Markdown | react-markdown + remark-gfm | AI 回复渲染 |
| AI 对话 | Fetch SSE 流式 | 兼容智谱 AI / OpenAI |
| 语音识别 | Web Speech API + 百度 WebSocket | 策略模式多 Provider |
| 语音合成 | Web Speech Synthesis API | 队列播放，流式朗读 |

### 1.3 核心亮点

1. **流式语音播放**：AI 回复边生成边朗读，体验流畅，是本项目最大的技术亮点
2. **零后端依赖**：纯前端实现，数据全部存本地，部署成本为零
3. **多语音引擎策略模式**：支持浏览器原生和百度语音识别切换，架构可扩展
4. **知识自动提取**：正则引擎从对话中自动识别 4 类结构化信息
5. **空对话优化**：新建对话不立即写库，首次发消息才持久化，避免数据冗余

## 二、系统架构

### 2.1 三层架构

项目采用清晰的三层分离架构：

```
┌─────────────────────────────────────────────────────────┐
│                     UI 组件层 (Components)                │
│   Sidebar │ MessageBubble │ KnowledgePanel │ Settings    │
├─────────────────────────────────────────────────────────┤
│                    状态管理层 (Stores)                    │
│     chatStore      │    voiceStore    │  knowledgeStore  │
├─────────────────────────────────────────────────────────┤
│                   服务层 (Services)                       │
│  AI Service │ Speech Service │ Storage Service │ Knowledge│
├─────────────────────────────────────────────────────────┤
│                   外部服务 / 浏览器 API                    │
│  智谱AI API │ Web Speech API │ IndexedDB │ localStorage  │
└─────────────────────────────────────────────────────────┘
```

**设计原则**：
- **组件层**只负责渲染和交互，不含业务逻辑
- **状态层**用 Zustand 集中管理运行时状态，桥接 UI 和 Service
- **服务层**封装所有 I/O 操作（网络请求、数据库、语音引擎），全部单例模式
- **可替换性**：Service 接口统一，比如语音识别可以切换不同 Provider

## 三、核心功能详解

### 3.1 AI 流式对话

**实现方式**：基于 SSE（Server-Sent Events）的流式输出

```
用户发送消息
  ↓
fetch API 请求（stream: true）
  ↓
ReadableStream + TextDecoder 逐块读取
  ↓
解析 SSE 格式，提取 delta.content
  ↓
逐字追加到 UI（打字机效果）
```

**为什么用 SSE 而不是 WebSocket？**
- 场景是"一问一答"，服务端单向推送，SSE 更轻量
- SSE 原生支持断线重连，不需要握手协议
- HTTP 代理和防火墙对 SSE 更友好

### 3.2 流式语音播放（核心亮点）

这是本项目技术含量最高的功能：**AI 回复边生成边朗读**。

**核心思路**：边生成 → 边切句子 → 边入队播放

```
AI 流式输出每个字符
  ↓
speakIfSentence() 检测完整句子
  ↓ 正则匹配句号/问号/感叹号
切出完整句子
  ↓
speakQueued() 加入播放队列
  ↓ stripMarkdown() 过滤格式
Web Speech API 自动排队播放
```

**关键技术点**：

| 技术点 | 实现方式 |
|--------|----------|
| 句子分割 | 正则 `exec()` 循环匹配，`spokenIndex` 记录播放位置 |
| 播放队列 | Web Speech API 原生队列 + `pendingCount` 计数器 |
| 代码块跳过 | `inCodeBlock` 状态机跟踪 ``` 边界，流式跳过代码内容 |
| Markdown 过滤 | 朗读前剥离加粗、标题、列表、代码块、Emoji 等格式符号 |
| 不阻塞 UI | `setTimeout(speakIfSentence, 0)` 放到下一个事件循环 |
| 语音选择 | 默认优先 Google 普通话，语速 1.2 倍 |
| 错误处理 | 忽略 `interrupted`/`canceled` 错误（停止播放的正常行为） |

### 3.3 语音识别（ASR）

采用**策略模式 + 管理器**设计，支持多 Provider 切换：

```
SpeechRecognitionManager（管理器）
  ├── Web Speech API（浏览器原生，免配置，默认）
  └── 百度语音（WebSocket + PCM 音频流，国内更稳定）
```

- 实时显示中间识别结果（`interimResults`），用户感知更快
- 网络错误时自动重连
- 预留阿里云、讯飞等 Provider 接口，架构可扩展

### 3.4 知识自动提取

从用户消息中自动识别 4 类结构化信息：

| 类型 | 触发模式 | 示例 |
|------|----------|------|
| 日程 (schedule) | 日期时间模式 | "明天下午3点开会" |
| 待办 (todo) | "提醒我"、"记得"、"待办：" | "提醒我明天买牛奶" |
| 笔记 (note) | "记住"、"记录"、"笔记：" | "记住这个API的用法" |
| 联系人 (contact) | 电话号码、邮箱模式 | "张三的电话是13800138000" |

提取的知识存入独立的 IndexedDB 数据库（`SoundAIKnowledge`），支持手动管理和搜索。

### 3.5 数据持久化

**分层存储策略**：

| 数据类型 | 存储方式 | 数据库/键名 | 说明 |
|----------|----------|-------------|------|
| 用户设置 | localStorage | `sound-ai-settings` | 含版本迁移机制 |
| API Key | localStorage | 独立键 | 加密存储 |
| 对话/消息 | IndexedDB | `SoundAIMessages` | Dexie，支持索引查询 |
| 知识库 | IndexedDB | `SoundAIKnowledge` | 独立数据库 |

**空对话优化**：
- 新建对话时 `currentConversationId = null`，仅存在于内存
- 发送第一条消息时才调用 `createConversation()` 写入数据库
- 避免产生大量空对话记录

## 四、数据流与核心流程

### 4.1 完整对话流程

```
用户输入文字 / 语音识别完成
  │
  ▼
App.handleSendMessage(content)
  │
  ├── 停止之前的语音播放
  ├── 确保有对话（首次发消息时持久化）
  ├── chatStore.addMessage(userMessage) ────── 写入 IndexedDB
  ├── chatStore.addMessage(空assistantMessage) ── 占位
  │
  ├── for await (chunk of aiService.chatStream())
  │     ├── 拼接 assistantContent
  │     ├── updateMessageContent() ── 仅更新内存（打字机效果）
  │     └── speakIfSentence() ── 按句分割，跳过代码块，队列朗读
  │
  ├── messageStorage.updateMessage() ── 流结束后更新数据库
  ├── generateTitleWithAI() ── 异步用 AI 生成精简对话标题
  └── knowledgeService.extractAndStore() ── 自动提取知识
```

### 4.2 状态管理设计

三个独立的 Zustand Store，各司其职：

| Store | 职责 | 关键状态 |
|-------|------|----------|
| `chatStore` | 对话管理 | conversations, currentConversationId, messages |
| `voiceStore` | 语音状态 | isListening, playbackState, voices, speed |
| `knowledgeStore` | 知识库 | items, selectedType, searchQuery |

**为什么选 Zustand 而不是 Redux？**
- 项目是中小规模，Zustand 更轻量（1KB）
- 不需要 Provider、reducer、action types 等样板代码
- 天然支持 selector 精确订阅，性能好
- 持久化和状态管理解耦（用 Dexie 而非 persist 中间件）

## 六、部署与运行

### 6.1 开发环境

```bash
# 安装依赖
npm install

# 启动开发服务器
npm run dev
```

### 6.2 生产构建

```bash
# 构建静态文件
npm run build

# 预览构建结果
npm run preview
```

构建产物在 `dist/` 目录，可部署到任意静态服务器。

### 6.3 部署方式

| 方式 | 适用场景 | 说明 |
|------|----------|------|
| Vercel / Netlify | 最简单 | 连接 GitHub，自动构建部署 |
| Nginx | 自有服务器 | 把 dist 目录放到 Nginx 静态目录 |
| ngrok | 临时测试 | 暴露本地端口，需配置 `allowedHosts: true` |

### 6.4 使用前提

1. 需要 AI 服务的 API Key（默认智谱 AI `glm-4-flash`）
2. 推荐使用 Chrome 浏览器（Web Speech API 支持最好）
3. 需要 HTTPS 或 localhost 才能使用语音功能

## 八、项目技术总结

| 维度 | 技术选型 | 设计考量 |
|------|----------|----------|
| 状态管理 | Zustand | 轻量、灵活、selector 精确订阅 |
| 本地存储 | Dexie (IndexedDB) | 大容量、结构化、异步、索引查询 |
| 流式输出 | Fetch + ReadableStream + SSE | 单向推送场景最优解 |
| 流式朗读 | 正则切句 + 原生队列 + 状态机 | 边生成边播放，不阻塞 UI |
| 语音识别 | 策略模式多 Provider | 可扩展、可替换 |
| 架构分层 | Components → Stores → Services | 关注点分离、可测试、可替换 |
| 类型安全 | TypeScript 严格模式 | 编译时错误检查、自文档化 |
